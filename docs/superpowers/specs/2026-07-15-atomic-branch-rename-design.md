# Atomic Branch Rename Metadata Design

## Problem

The manual branch-rename, pre-push, and pre-PR interceptors currently persist a renamed branch with two sequential workspace updates: one sets `branchName`, and the next clears `isAutoGeneratedBranch`. If the second update fails, the database retains the user's chosen name with the auto-generated flag still set. A later pre-push or pre-PR hook then treats that user branch as generated and replaces it.

## Considered Approaches

1. Add one workspace-data operation that writes `branchName` and `isAutoGeneratedBranch: false` in a single accessor update. This is the recommended approach because it enforces the invariant with one Prisma statement and uses the existing service-capsule boundary.
2. Wrap the two current updates in a Prisma transaction. This would be atomic but requires transaction-aware plumbing through the accessor and service for a two-field update that Prisma already supports directly.
3. Add defensive branch-name detection to pre-push and pre-PR. This treats the symptom, depends on naming heuristics, and cannot reliably distinguish a user-selected branch while the persisted flag is wrong.

## Design

Add `workspaceDataService.setBranchNameAndClearAutoGenerated(id, branchName)`. It will call `workspaceAccessor.update` once with both `branchName` and `isAutoGeneratedBranch: false`. The three interceptors will use this method after a successful Git rename or manual rename detection.

The existing `clearAutoGeneratedBranch` method remains for the pre-push and pre-PR paths where the generated name already equals the stored branch name. Those paths do not mutate `branchName`, so they do not have a two-field atomicity requirement. No Prisma schema, migration, service registry, or barrel export changes are needed.

## Error Handling and Edge Cases

- If the combined database update fails, neither metadata field is persisted by that update, eliminating the partial database state reported in issue #1843.
- Failed or empty Git branch lookups and failed Git renames continue to skip persistence.
- Non-auto-generated branches continue to bypass pre-push and pre-PR rename logic.
- Git rename followed by total database failure can still leave Git and database branch names divergent. That pre-existing cross-system consistency case is outside this issue, which concerns partial persistence within the database update.

## Testing

- Update each interceptor test to require the combined service method on successful rename paths and to retain existing negative-path assertions.
